# 今日修改总结

本文档总结今天针对 star-searcher 迁移版探索流程所做的分析、修正和当前结论，重点围绕以下几个实际运行问题：

- 飞机在局部区域反复绕圈，频繁重规划。
- 重规划目标经常落回 `local_tour_[0]` 或已经接近/到达的局部 viewpoint。
- 运行一段时间后出现 `path too short, replanning`、`plan fail`、`No path to next viewpoint`。
- 偏航角在部分时刻出现明显不合理的大幅跳变，甚至导致大绕圈。
- 飞出已探索区域后，下一次规划又飞回已充分探索区域；长时间运行后甚至回到最开始的旧区域并持续绕圈。

## 1. 总体结论

今天的修改分为四条主线：

1. 修正局部 tour 与 TSP/索引之间的错位问题，避免规划总是落回错误的局部目标。
2. 修正 yaw 目标选择和重规划时机，减少由于过早 replan 和角度包裹导致的绕圈。
3. 改善局部 viewpoint 的选择、可达性判断和 fallback 策略，降低 `No path`、短路径死循环、原地打转问题。
4. 最新一轮开始从 frontier 生命周期入手，统一 frontier 的“生成/保留/删除/覆盖完成”判定，尝试根治“回飞已探索区域”的问题。

其中，前 3 条主线已经完成代码落地并通过编译；第 4 条主线今天也已完成第一版代码修改并通过编译，但还需要运行验证其是否足以压制“旧 frontier 残留吸引规划回跳”的现象。

## 3. 详细修改说明

### 3.1 `fast_exploration_manager.cpp`

这是今天改动最多的文件，主要处理“下一目标怎么选”“路径怎么搜”“yaw 怎么跟”“何时跳过当前局部目标”这四件事。

#### 3.1.1 修正局部 tour 目标选择

主要问题：

- 之前局部 planning 容易反复落在已经接近的 checkpoint 上。
- `local_tour_` 相关逻辑会让飞机不断 replan 到过近甚至已达到的 viewpoint。

主要修改：

- 不再盲目使用第一个局部 checkpoint，而是结合局部 TSP 结果筛选真正还没到达的候选点。
- 当所有局部 viewpoint 都已经很近时，不立即继续追逐局部点，而是优先考虑切到 `next_cluster_pos`。
- 单个局部 viewpoint 已经达到时，也不再直接陷入失败，而是尝试切换到下一 cluster 中心。

效果：

- 减少了“重复追逐同一个局部点”的情况。
- 降低了 `Single local viewpoint already reached -> force frontier update -> 再次回到相同区域` 这种小循环概率。

#### 3.1.2 增加局部目标 fallback 与路径长度判定

主要问题：

- 某些局部 viewpoint 虽然在排序上靠前，但实际上不可达。
- 某些局部 viewpoint 虽然可达，但路径极短，导致轨迹刚起飞就再次触发 replan。

主要修改：

- 引入 `local_candidates`，依次测试多个局部 viewpoint，而不是只看一个。
- 对每个候选点先做 A* 路径搜索，若不可达则跳过。
- 若所有局部路径都很短，则保留“最不短”的那个作为退化选择。
- 当局部路径很短，但下一 cluster 中心明显更有意义时，优先改飞 cluster center。

效果：

- 减少 `No path to next viewpoint`。
- 降低局部短路径导致的反复触发 replanning。
- 在局部 viewpoint 质量差时，能更平滑地退回 cluster 级目标。

#### 3.1.3 增加 A* 搜索端点修复

主要问题：

- 飞机在某些时刻靠近 inflated obstacle、边界或局部非理想位置，A* 直接从当前位置或目标点起搜会失败。

主要修改：

- 新增 `point_is_searchable` 与 `project_to_searchable`。
- 在 A* 前先检查起点和终点是否适合作为搜索端点。
- 若不适合，则在附近搜索可用 free 点作为临时搜索端点。
- 搜索成功后再把原始位置补回路径头尾。

效果：

- 减少因端点不合法导致的假性 `No path`。
- 对轻微“卡边”或贴近膨胀障碍的场景更鲁棒。

#### 3.1.4 修正 yaw 目标选择

主要问题：

- 以前的 yaw 选择直接取 `min_yaw` 或原始候选值，可能跨越 $\pm\pi$ 边界，导致看起来像 360° 大转头。

主要修改：

- 新增 `normalizeAngle` 和 `nearestEquivalentYaw`。
- 所有候选 yaw 先变换到“距离当前 yaw 最近的等价值角度”再比较。
- `planYawExplore` 改为真正使用选出的 `target_yaw`，并启用前视约束。

效果：

- 明显降低了由角度包裹引起的假性大幅转头。
- 减少了“轨迹没问题但机头突然大回摆”的情况。

#### 3.1.5 针对短路径的 yaw 策略调整

主要问题：

- 当局部 motion 很短时，如果还严格对着局部 viewpoint 摆头，容易出现小范围绕圈或视觉上的摇摆。

主要修改：

- 当路径过短且下一 cluster 更远时，让 yaw 对准 `next_cluster_pos`，而不是只盯住当前局部小点。

效果：

- 降低短局部动作中的无意义摆头。
- 更有利于继续向未知区域推进。

#### 3.1.6 修正 `findNextCluster` 的 cluster 跳过逻辑

主要问题：

- 原先默认直接从 `indices[0]` 开始，很容易继续停留在已经基本走完的当前 cluster。

主要修改：

- 遍历 cluster 顺序，检查当前 cluster 的全部 checkpoint 是否已经接近当前位置。
- 如果当前 cluster 实际上已经被“走到”，则切到后续 cluster。
- `getCheckTour` 调用也改为传入 `cur_pos` 和 `next_cluster_pos`，让 checkpoint 选择与实际运动方向相关。

效果：

- 减少“离开当前 cluster 后又被拉回当前 cluster”的情况。

### 3.2 `fast_exploration_fsm.cpp`

这个文件主要改的是 replanning 触发时机。

主要问题：

- 之前 `cluster covered` 和 `periodic call` 在轨迹还没执行多少时就触发，导致短轨迹被频繁打断，飞机表现为局部绕圈。

主要修改：

- 新增 `exec_ratio`，用轨迹执行比例辅助判断是否该重规划。
- `traj fully executed` 不再只用固定阈值，而是根据轨迹时长动态计算 `end_replan_thresh`。
- `cluster covered` 触发被收紧到轨迹末段。
- `periodic call` 也增加执行比例约束，避免过早打断。

效果：

- 降低“轨迹刚发出去又被新规划打断”的问题。
- 减少局部路径尚未真正执行就重新算目标导致的打转。

### 3.3 `frontier_finder.cpp` 与 `frontier_finder.h`

这是今天诊断的重点之一，也是最新一轮修正“回飞已探索区域”的核心。

#### 3.3.1 修正 `getCheckTour` 的 viewpoint 选择方式

主要问题：

- 旧逻辑基本等价于固定取 `viewpoints_.front()`。
- 这会让局部 checkpoint 更偏向“视角最好”的点，而不一定是“当前最该去、最易去、最利于继续往下一 cluster 走”的点。

主要修改：

- `getCheckTour` 现在显式接收 `cur_pos` 与 `next_cluster_pos`。
- 对每个 frontier 的 viewpoint 建立评分，综合考虑：
  - 与当前位置距离
  - 是否可达
  - 是否沿着前进方向推进
  - 可见数量
  - 是否有利于离开当前 cluster 朝下一个 cluster 继续移动
- 选择评分最高的 viewpoint 作为 checkpoint。

效果：

- 局部 checkpoint 更贴近实际推进方向。
- 减少“为了一个观测姿态优但方向反了的 viewpoint 而折返”的情况。

#### 3.3.2 修正 checkpoint 间 yaw cost

主要问题：

- checkpoint 之间的 yaw cost 原先是简单减法，没有做角度 wrap 和绝对值处理。

主要修改：

- 改为使用最小包裹角差：

$$
\Delta \psi = \min(|\psi_1-\psi_2|, 2\pi - |\psi_1-\psi_2|)
$$

效果：

- 局部 TSP 的代价更符合真实转头成本。
- 减少被错误 yaw cost 牵引出的不合理次序。

#### 3.3.3 `isFrontierCovered()` 只看 active frontier

主要问题：

- 以前 `cluster covered` 的判定会参考全部 frontier，导致当前 cluster 的 replan 条件被其他无关 frontier 干扰。

主要修改：

- 记录当前 `check_tour` 对应的 `active_frontier_ids_`。
- `isFrontierCovered()` 只检查这些 active frontier。

效果：

- 覆盖完成判断更聚焦当前正在执行的 cluster。
- 减少误触发 `cluster covered`。

#### 3.3.4 统一 frontier 有效性判定

这是今天最后新增、最关键的一项修正。

发现的问题：

- frontier 的生成条件是：`knownfree && (neighbor underobserved || neighbor unknown)`。
- 但 frontier 的删除条件长期只按：`knownfree && neighbor unknown` 的反条件判断。
- 这意味着 frontier 的“生成语义”和“保留/失效语义”并不一致。
- 结果是：某些已经不应再吸引规划的 frontier 可能继续留在 `frontiers_` 或 `dormant_frontiers_` 里，后面又被重新规划回来。

主要修改：

- 新增统一判定函数 `isFrontierCellActive()`。
- `searchFrontiers()` 中新 frontier 的种子判定改用这个统一函数。
- `isFrontierChanged()` 也改用同一函数判断旧 frontier 是否已经失效。
- `isFrontierCovered()` 同样改用这套规则。

效果：

- 生成、保留、覆盖完成三处逻辑统一。
- 为抑制“旧 frontier 长期残留，导致飞机回飞已探索区域”提供了更一致的基础。

#### 3.3.5 更积极地清理旧 frontier 与 dormant frontier

发现的问题：

- 原逻辑删除 frontier 时依赖“frontier box 与 updated box 重叠”且 `isFrontierChanged()` 为真。
- 这会导致一个现实问题：即使某块区域已经被充分探索，只要它没有恰好落在当前 update box 删除逻辑里，旧 frontier 也可能继续存活。

主要修改：

- 现在在 `searchFrontiers()` 中，对已有 `frontiers_` 和 `dormant_frontiers_` 都执行一次更主动的失效检查，不再严格依赖 overlap 才允许删除。

效果：

- 旧 frontier 更容易及时退役。
- 理论上有助于减少后续规划又跳回旧区域。

说明：

- 这是今天最新的根因向修正，代码已落地并编译通过，但仍需运行日志进一步验证效果。

### 3.4 `planner_manager.cpp`

主要问题：

- yaw 快速变化告警基于原始角度差，存在 unwrap 后显示不准确的问题。

主要修改：

- 使用 unwrap 后的 `yaw_delta` 做 rapid change 检查。

效果：

- 调试信息更可信，便于判断是真跳变还是角度表示问题。

### 3.5 `algorithm.xml`

主要修改：

- `frontier/min_view_finish_fraction` 从 `0.2` 提高到 `0.45`。

目的：

- 让 frontier 被视为“完成”的门槛更高一些，降低 frontier 过早被判断为处理完、随后又被重新吸回来的概率。

说明：

- 这个参数属于经验性调节，不是根因修复本身；真正的根因修正仍然在 frontier 生命周期一致性上。

## 4. 今天已经确认修掉或明显缓解的问题

截至目前，以下问题已经有明确修正或明显缓解：

- `local_tour_` 与实际候选索引错位导致的错误局部目标选择。
- 已经接近/达到的局部 viewpoint 仍被不断重复选择。
- 局部目标不可达时直接失败而没有 fallback。
- 由 $\pm\pi$ 包裹引起的 yaw 目标误判和大转头。
- 轨迹刚执行不久就被 `cluster covered`/`periodic call` 打断。
- A* 起终点局部不可用导致的部分假性 `No path`。

## 5. 当前最值得继续观察的问题

今天虽然已经开始从 frontier 生命周期根因修复，但以下两类问题还需要继续用仿真日志验证：

1. 飞机是否还会从已经充分探索的区域重新飞回去。
2. 长时间飞行后，是否还会回到最早的旧区域并持续绕圈。

如果这两个问题仍明显存在，下一步最可能继续深挖的方向是：

1. `dormant_frontiers_` 的重新激活/继续保留机制是否仍偏宽松。
2. cluster 级代价与排序是否对“刚刚离开的旧区域”缺少时间上的抑制。
3. 是否需要增加“recently visited / recently completed region suppression”一类的短时去重机制。
